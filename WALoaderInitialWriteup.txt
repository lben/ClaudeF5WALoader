# AGENTS.md

## Purpose

This repository is for WALoader, a Streamlit-based internal app platform for finance users who are not software developers but can work with LLM coding agents.

WALoader lets users upload LLM-generated Streamlit projects as structured markdown bundles, reconstructs those bundles into real Streamlit projects, validates them, deploys them, manages their lifecycle, maps uploaded datasets to named Dataset Concepts, and optionally provides reusable user-management support for each deployed app.

The target apps are usually low-to-medium complexity finance apps focused on loading, visualizing, and analyzing data. Most apps are read-only dashboards or analysis tools. Some apps may allow limited writes, mainly for configuration-like data.

The primary objective is to make deployment and management of LLM-generated finance Streamlit apps as automated, safe, and low-friction as possible.

## Non-negotiable operating rules for agents

- Do not read, list, search, summarize, modify, move, delete, or infer anything under `/private`.
- Do not generate code that reads from or writes to `/private`.
- Do not include `/private` in backups, logs, searches, markdown bundle parsing, dataset handling, Caddy configuration, process management, or cleanup.
- Do not hardcode secrets, tokens, passwords, package repository credentials, private certificate material, or user-specific credentials.
- Do not commit private uv, pip, Caddy, or corporate artifact repository credentials.
- Use small, milestone-based implementation.
- Do not build the UI first.
- Do not implement broad future architecture that is not needed for the current milestone.
- Prefer automation over manual operator instructions.
- If automation and manual instructions conflict, implement automation first and document the manual fallback second.
- Do not stop after partial scaffolding if the requested milestone can be completed.
- If blocked, document the blocker clearly in `DEVLOG.md` with exact error text, files changed, validation attempted, and next recommended action.
- Do not weaken tests merely to make tests pass.
- Do not add undocumented configuration settings.
- Keep the system as simple as possible while still satisfying the product requirements.

## Model and reasoning expectations

Use the highest-quality model and reasoning effort selected by the user in the IDE.

Do not assume access to a specific model name. The user may select Claude Opus High or another high-capability model in VS Code Agent Mode.

Operate in careful high-reasoning mode:

- understand the goal before editing;
- plan before coding;
- implement one milestone at a time;
- test service-layer behavior;
- update project memory files after every implementation session;
- prefer explicit acceptance criteria over vague progress.

- summary;
- files changed;

- next recommended action.

Do not rewrite historical `DEVLOG.md` entries except to fix formatting.

## Implementation workflow

Use this order:

1. Plan milestone.
2. Confirm scope against the active goal file.
3. Implement service-layer functionality first.
4. Add or update tests for behavior changed in the milestone.
5. Run validation.
6. Update `PROGRESS.md`.
7. Append to `DEVLOG.md`.
8. Stop only after the milestone is complete or a concrete blocker is documented.

For implementation prompts that say "implement milestone Mxx", complete the milestone in the current session. Do not stop after analysis only. Do not expand beyond the milestone scope.

## Build order

Build WALoader in this order unless the user explicitly changes the plan.

This order is intentionally dependency-first. Foundational utilities, configuration, database access, repositories, and service-layer modules must be implemented before UI clients or orchestration clients that depend on them.

The agent must not build a dependency client before the dependency exists, except for a small explicit interface stub that is documented in the active milestone goal file and replaced before the feature is considered complete.

## Dependency rule

Each milestone must explicitly list its dependencies.

A milestone must not implement a client of a service before that service exists, unless the milestone creates a small documented interface stub for that service. Stubs must be replaced before the feature is considered complete.

Examples:

- The dashboard must not call deployment behavior before the deployment service exists.
- The deployment service must not call Caddy behavior before the Caddy manager exists.
- The deployment service must not allocate ports before the port allocation service exists.
- Health checks must not send crash emails before the notification service exists.
- Dataset UI must not call Dataset Concepts behavior before Dataset Concepts services exist.
- App user-management UI must not call app user-management behavior before app user-management services exist.

Prefer bottom-up implementation for services and vertical-slice validation after each major service group.

## Product user roles

WALoader has at least these user roles:

- WALoader admin: the operator/admin of the WALoader platform.
- App owner: a WALoader user who uploads and manages one or more child apps.
- App user: a user who opens and uses a deployed child app.
- Anonymous/unauthenticated visitor: a user who has not logged in yet.

The WALoader admin can access the admin panel and platform configuration. App owners can manage their own apps and, when user-management support is enabled, manage users for their own apps.

## Product user flow

The intended user flow is:

1. A finance user works independently with a coding LLM to build a Streamlit app.
2. The finance user asks the coding LLM to provide a structured markdown file containing the entire project, except binaries or large files that should be attached separately.
3. The user loads the WALoader site and logs in.
4. The user dashboard shows the apps owned by that user.
5. The dashboard includes a "Create new app" button.
6. When "Create new app" is selected, the user can:
   - upload the markdown project bundle;
   - enter an app name;
   - enter an app description;
   - enable or disable "Users Management Support";
   - define Dataset Concepts if needed.
7. While the user types the app name:
   - WALoader generates a URL label / slug from the name;
   - WALoader checks SQLite to see whether the app name or slug is already taken;
   - the UI shows a check mark when available;
   - the UI shows an X when unavailable;
   - if unavailable, the UI shows a smaller italic message under the name field explaining that the name is already taken.
8. When the user selects Submit:
   - WALoader sends the markdown bundle to the backend service layer;
   - WALoader validates the bundle;
   - WALoader reconstructs the project into a versioned source directory;
   - WALoader runs tests if the uploaded app includes tests;
   - WALoader finds an available port;
   - WALoader creates or updates the app runtime;
   - WALoader starts the child Streamlit app;
   - WALoader updates Caddy routing if Caddy is enabled;
   - WALoader performs initial health checks.
9. On successful deployment, WALoader shows a success screen or modal with:
   - a success message;
   - a copyable code block containing the app URL;
   - a link to the deployed app, opening in a new tab if Streamlit allows it.
10. On deployment failure, WALoader shows an error screen or modal with:
   - a concise failure message;
   - a copyable code block containing concatenated deployment, test, dependency, and runtime errors;
   - an upload button that allows the user to upload a fixed markdown bundle and retry.
11. The same upload/deploy flow must be composable and reused for:
   - initial app creation;
   - retry after failed creation;
   - app code update;
   - retry after failed update.

## User management support module

WALoader must provide a reusable user-management module.

The module is enabled by default for WALoader itself. App owners may enable or disable it for their own apps.

If a child app has user-management support enabled:

- app users must log into that child app before using it;
- the child app must show login/logout support;
- the child app must support password change;
- unauthenticated users should see the login form as the default screen.

The user-management module must support:

- table of existing users;
- create user;
- update user;
- deactivate user;
- reactivate user;
- delete user;
- change password;
- user email address;
- observations text area;
- dynamic attachment metadata and files, such as screenshots or other evidence provided to justify access.

The module should be implemented as a central WALoader service plus reusable SDK/helper logic, not copied into each child app.

Generated child apps may use:

```python
from waloader_sdk.auth import require_login
```

If user-management support is disabled for a child app, app-level login is not required for that child app.

Do not store plaintext passwords. Use a secure password hashing library such as `argon2-cffi`.

## Dataset Concepts module

WALoader must provide each app with the ability to define and upload datasets associated with named Dataset Concepts.

Dataset Concepts are stable concept names that the generated app code expects, such as:

- clients
- transactions
- positions
- accounts
- trades
- reference_data

The user may tell the coding LLM that the app will use a concept by referring to the concept in brackets, such as:

- `[clients]`
- `[transactions]`
- `[positions]`

Generated app code should use the WALoader SDK:

```python
from waloader_sdk.datasets import load_dataset

clients = load_dataset("clients")
```

If no file has been uploaded for a concept, the app should show an empty space with an italic message:

No data uploaded yet

Dataset upload support must include:

- `.csv`
- `.xlsx`
- `.xls`
- `.parquet`

CSV is the default expected format, but Excel and parquet are supported.
If an Excel file (`.xlsx` or `.xls`) is uploaded for a Dataset Concept, WALoader must require a sheet name. The upload UI must show a sheet-name input pre-filled with `Sheet1`. The sheet name is required for Excel files and must be stored with the dataset file metadata so the app can load the correct sheet later. CSV and parquet uploads do not require a sheet name.

## Dataset Concepts flow

The Dataset Concepts flow must support:

1. When creating or editing an app, the user can open the Dataset Concepts mapping screen.
2. The screen shows a text field for a new concept name.
3. The user enters a concept name, for example `clients`.
4. The user selects save.
5. The saved concept appears in the concept list.
6. Each saved concept row includes:
   - concept name;
   - upload file button;
   - delete button.
7. Deleting a concept requires confirmation.
8. Uploading a file automatically stores the file and associates the file with the concept.
9. For an existing concept with an existing uploaded file, the user can upload a replacement file.
10. Replacement upload compares old and new schemas:
    - headers;
    - inferred data types;
    - added columns;
    - removed columns;
    - changed column types.
11. If schema discrepancies exist:
    - report the discrepancies to the user;
    - show the schema diff in a copyable code block;
    - ask for confirmation before overwriting.
12. A schema mismatch may be acceptable if the user has also updated the app code to use the new schema.

The schema diff feature must be implemented in a reusable service function and tested.

For Excel uploads:

- when the selected file extension is `.xlsx` or `.xls`, show a sheet-name input;
- pre-fill the sheet-name input with `Sheet1`;
- require the sheet name before saving the upload;
- store the sheet name with the dataset file metadata;
- use the stored sheet name when reading the Excel file;
- include the sheet name in schema inference and schema comparison operations.

For CSV and parquet uploads, do not ask for a sheet name.

## Markdown project bundle contract

WALoader accepts structured markdown project bundles.

Bundle requirements:

- metadata section;
- file blocks with relative paths;
- declared entrypoint;
- optional `pyproject.toml`;
- optional tests;
- no binaries embedded in markdown;
- binary files must be attached separately if needed.

Parser safety rules:

- reject absolute paths;
- reject `..`;
- reject paths under `/private`;
- reject paths that escape the app root;
- reject dangerous hidden paths unless explicitly allowed;
- never overwrite WALoader source files;
- reconstruct into a versioned app source directory;
- preserve file contents exactly inside fenced blocks where practical.

Every upload creates a new app version.

The upload/deployment service must be composable so creation, retry, update, and update retry reuse the same backend logic.

## App slugs

An `app_slug` is a safe URL and filesystem identifier derived from the user-facing app name.

Examples:

```text
"Client Positions Dashboard" -> "client-positions-dashboard"
"P&L Explain 2026!" -> "p-l-explain-2026"
```

The slug is used in:

```text
URL:
http://servername:8080/apps/client-positions-dashboard

Folder:
data/apps/client-positions-dashboard/

Logs:
data/logs/apps/client-positions-dashboard/

Database:
apps.slug = "client-positions-dashboard"
```

Rules:

- lowercase;
- letters, numbers, and hyphens only;
- spaces and punctuation become hyphens;
- repeated hyphens collapse into one;
- no leading or trailing hyphen;
- max length 60 characters;
- must be unique;
- reserved names are blocked.

Reserved names include:

```text
waloader
admin
login
logout
api
static
assets
private
health
apps
app-link
caddy
```

## Configuration philosophy

Keep configuration simple.

Only require settings that cannot be safely inferred. Prefer derived defaults over required configuration.

Primary required or near-required settings:

- Python binary path
- uv binary path
- Caddy binary path if Caddy support is enabled
- uv config file path if the environment requires a private package index
- public host if clean URLs should use a specific server name

All other paths should derive from `paths.data_dir` unless explicitly overridden.

Example derived paths:

```text
database_path      = data/waloader.db
apps_dir           = data/apps
logs_dir           = data/logs
backups_dir        = data/backups
uploads_dir        = data/uploads
tmp_dir            = data/tmp
uv_cache_dir       = data/uv-cache
caddy_config_path  = data/caddy/Caddyfile
caddy_logs_dir     = data/logs/caddy
```

Every configurable setting must be documented in:

1. `config/waloader.example.toml` using TOML comments where practical.
2. `docs/configuration.md`.

For every setting, document:

- setting name;
- purpose;
- default;
- required or optional status;
- allowed values;
- what happens when each allowed value is used;
- when to change it;
- what happens if missing;
- what happens if wrong;
- whether it may contain secrets;
- whether it is safe to commit.

Do not add undocumented settings.

## uv and dependency installation

WALoader assumes the operator has configured uv so it can install packages from the approved company package index before running deployments.

Do not rely on `pip.ini` for uv behavior. uv uses uv-native configuration.

WALoader must support:

```toml
[executables]

[uv]

ssl_cert_dir = ""
```

When invoking uv:

- pass `UV_CONFIG_FILE` if `uv.config_file` is set;
- pass `UV_CACHE_DIR` if `uv.cache_dir` is set;
- pass `UV_SYSTEM_CERTS=true` if `uv.system_certs` is true;
- pass `SSL_CERT_FILE` if configured;
- pass `SSL_CERT_DIR` if configured;
- use the configured Python executable through `--python <path>` when `executables.python_binary` is set;
- pass host-specific insecure settings only when explicitly configured;
- preserve existing environment variables unless intentionally overridden;
- never read, print, or log the contents of `uv.config_file`;
- never log secrets or token-bearing URLs.

Default preflight:

```bash
uv pip install --dry-run pandas
```

Preflight should use the same uv binary and environment that deployments will use.

If dependency install fails, WALoader must show a clear, copyable error block and log the full technical details without exposing secrets.

## Dependency policy

Use per-app virtual environments for deployed child apps, with a shared uv cache directory.

Do not use one global mutable venv for all child apps.

Do not rely on uv inline script metadata for deployed multi-file Streamlit apps. uv scripts may be used for small maintenance tools only if useful.

Recommended defaults:

```toml
[dependencies_policy]
allow_app_dependencies = true
allow_direct_url_dependencies = false
allow_vcs_dependencies = false
allow_path_dependencies = false
require_admin_approval_for_new_dependencies = false
```

Meaning:

- `allow_app_dependencies = true`: child apps may declare normal package dependencies in `pyproject.toml`.
- `allow_app_dependencies = false`: child apps may only use the WALoader-approved base dependency set.
- `allow_direct_url_dependencies = true`: allow dependencies from direct wheel/source URLs.
- `allow_direct_url_dependencies = false`: reject direct URL dependencies.
- `allow_vcs_dependencies = true`: allow dependencies from Git or other VCS URLs.
- `allow_vcs_dependencies = false`: reject VCS dependencies.
- `allow_path_dependencies = true`: allow local path dependencies.
- `allow_path_dependencies = false`: reject local path dependencies.
- `require_admin_approval_for_new_dependencies = true`: block new app dependencies until admin approval.
- `require_admin_approval_for_new_dependencies = false`: attempt install automatically and report failures.

Reject dependency forms that violate policy with clear user-facing errors.

## Caddy reverse proxy

Caddy is the preferred reverse proxy.

Do not assume root access. Do not assume Caddy is installed globally. Do not assume port 80 is available.

Caddy binary path must be configurable before WALoader starts.

Default public Caddy port should be `8080`.

Preferred clean URLs:

```text
http://servername:8080/waloader
http://servername:8080/apps/<app_slug>
```

Fallback direct URLs:

```text
http://servername:<allocated_child_port>
```

WALoader must provide service-layer Caddy operations and expose them through CLI and admin UI:

```bash
uv run python -m waloader.tools.caddyctl generate
uv run python -m waloader.tools.caddyctl validate
uv run python -m waloader.tools.caddyctl start
uv run python -m waloader.tools.caddyctl stop
uv run python -m waloader.tools.caddyctl reload
uv run python -m waloader.tools.caddyctl status
```

WALoader should generate the Caddyfile from the app database. Users should not need to hand-edit it during normal operation.

If Caddy is enabled and available:

- WALoader launches itself with `--server.baseUrlPath=waloader` when appropriate.
- Child apps launch with `--server.baseUrlPath=apps/<app_slug>`.
- Caddy routes `/waloader` to the WALoader port.
- Caddy routes `/apps/<app_slug>` to each child app port.

If Caddy fails, WALoader should fall back to direct port URLs and report the Caddy error clearly.

## Notifications and crash emails

WALoader must distinguish deployment failures from production crashes.

Do not send crash notification email for:

- dependency installation failure;
- test failure;
- initial deployment failure;
- user-triggered stop;
- user-triggered restart;
- app update failure.

Send crash notification email only when:

- the app was previously running;
- it passed initial health checks;
- it lived beyond the configured production grace period;
- a later health check detects process death or repeated unhealthy status;
- state transitions from `running` to `failed`;
- no email has already been sent for that failure event.

Use a notification adapter module. Do not scatter email calls across services.

The available email function has this signature:

```python
send_mail(
    subject=email_subject,
    sender='[sender_email]',
    recipients=recipients_list,
    html_body=outlook_compatible_email_html_content,
)
```

The notification module should call this adapter when configured.
